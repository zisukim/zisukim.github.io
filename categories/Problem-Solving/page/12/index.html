재귀함수 호출(Recursive function call) : 함수가 자기 자신을 호출!

  1. 내가 하려는 작업이 있다
  2. 문제를 잘게 자른다.
  3. 더 이상 자를 수 없는 부분(base) 처리

base case로 더 이상 쪼갤 수 업는 부분을 처리하고
recursive case로 쪼갠 문제를 호출한다.

1182 부분집합의 합

https://www.acmicpc.net/problem/1182

input : 원수의 개수 1<=N<=20
부분집합 원소의 합 S

원소의 개수가 최대 20개일 경우 부분집합 원소의 개수는 $2^20$(1,049,576)개이다.
완전탐색으로 모든 경우의 수를 구한 후 S와 같은 경우의 수를 구할 수 있다.
원소의 값들을 저장한 배열의 index를 인자로 하여 0번 index부터 순차적으로 살펴볼검

arr[index] 원소를 sum에 포함시키는 경우 포함을 시키지 않는 경우

함수 : n번째 원소를 인자로 호출한다.
인자 : 원소의 값을 저장한 배열의 index
base case : index가 N이 되었을 때 더 이상 쪼개지지 않은 작업이 된다.
내가 하려는 작업 : sum에 원소를 포함시킨다, sum에 원소를 포함시키지 않는다.
recursive case : index + 1번째 원소를 인자로 호출한다.

공집합인 경우를 제외하므로 S가 0일 때는 경우의 수 중 한 개를 빼준다.
(S=0일때 공집합일 경우에도 sum == S가 성립하므로)

참고용 : integer return 버전의 재귀함수


4963 섬의 개수

https://www.acmicpc.net/problem/4963

현재 우리가 구현하고 있는 재귀는 DFS(Depth first search : 깊이 우선 탐색) 모양을 가진다.
갈 때까지 가보자는 알고리즘이다. 첨부된 ppt(부분집합의 합 분제)에서 나타나 있는 트리가 맨 마지막 노드까지 내려가는 것처럼!
(뭔말인지 모르겠져 자세한 설명은 그래프 시간에ㅎㅎ)

DFS로 세가지 사실을 알 수 있다.

  1. 탐색
  2. 연결성
  3. component 개수
    (뭔말인지 모르겠져 자세한 설명은 그래프 시간에2ㅎㅎ)

이 문제 같은 경우 component의 개수를 세는 문제이다.
component는 서로 연결되어 있는 뭉탱이로 생각하면 된다.

이 경우 component는 2개이다.
여튼 이 문제에서 이제 재귀함수를 어떻게 구현해야 할지 생각해보자.
재귀함수를 호출하는 것을 방문한다라고 생각을 한다.
그 다음 우리가 다음에 방문하고 싶은 곳을 recursive case에서 인자로 설정하여 방문한다고 생각하면 된다.

함수 : map[y][x]를 방문한다.
인자 : 현재 방문할 y, x
base case : main에서 지도 끝까지 뒤졌을 때 끝난다.(함수 안에는 구현안됨)
내가 하려는 작업 : map[y][x]을 방문한다. visited[y][x]에 방문했다는 표시를 해준다.
recursive case : 8방향을 살펴본 뒤 조건에 맞으면 다음 y, x인자를 설정하여 함수를 호출한다.

recursive case에서 다음 인자를 선택할 조건

nextY를 다음에 방문할 y좌표, nextX를 다음에 방문할 x좌표라고 생각한다.
① map[nextY][nextX]가 육지이다.
② map[nextY][nextX]를 아직 방문하지 않았다. (visited[nextY][nextX]의 값이 0이다.)

long long s = $w*h$;

이런 연산을 한다고 할 경우

그렇지 않으면 int형 변수로 overflow난 다음에 long long타입의 s변수에 들어가기 때문에 다른 값이 저장된다.

빌딩

Posted on 2015-04-15 08:53 | In Problem Solving | Comments:

Problem

http://www.jungol.co.kr/bbs/board.php?bo_table=pbank&wr_id=607&sca=30

Solution

$n^2$으로 불가능한 input size
입력받으면서 풀기 잼

  1. i<j && Height of i<Height of j
  2. 1의 조건을 만족하면 j+1부터 아예 볼 수조차 없음

즉 i보다 큰 빌딩(j) 한 개를 만나면 그 뒤의 빌딩(j+1이후)은 앞에서 만난 큰 빌딩(j)때문에 가려서 안보임

모든 input을 받고 비교하는 것보다
필요한 input만을 저장하여 해결하자.

stack으로 풀기

stack안에 빌딩의 높이들을 하나씩 추가한다.
새로 입력되는 빌딩의 높이를 n이라고 하자.
stack의 top보다 n이 작다면 아직 top은 볼 수 있는 빌딩(top보다 높은 빌딩)을 만나지 못했으므로 push한다.
stack의 top보다 n이 크다면 top은 볼 수 있는 빌딩(top보다 높은 빌딩)을 만났다.
top의 빌딩에서 볼 수 있는 가장 큰 빌딩의 높이는 n인 것이다.
이는 문제에서 top의 답이므로 더이상 top은 앞으로의 연산에 필요하지 않다.
top을 pop시킨다.
top을 pop시킨 것이 바로 top의 빌딩에서 볼 수 있는 가장 큰 빌딩이다.
pop시킨 빌딩의 인덱스를 당연히 저장시킨다. 답이니깐 ㅎㅎㅎ

top이 바뀌었다.
다음 top은 아직 볼 수 있는 빌딩(top보다 높은 빌딩)을 만나지 못한 빌딩이다
똑같이 계속 연산을 해 준다.

불쾌한 날도 그렇고 빌딩도 그렇고 더이상의 연산에 필요가 없을 때 pop을 해준다.
