재귀함수2

Posted on 2015-05-06 13:43 | In Problem Solving | Comments:

재귀함수에 대한 전반적인 이론은 재귀함수1 참조
재귀함수1에서 풀었던 두 문제 1182 부분집합의 합 4963 섬의 개수와 솔루션이 비슷한 문제들

주사위 던지기2

http://www.jungol.co.kr/bbs/board.php?bo_table=pbank&wr_id=458&sca=20

경우의 수를 더해서 조건을 만족하면 출력하는 점에서 부분집합의 합과 매우 비슷한 문제이다.
다만, 부분집합의 합과 다른 점은 다음 과 같다.

부분집합의 합

  • 재귀 : 해당 원소를 포함한 경우 하지 않는 경우
  • 인자 변화 : 배열의 index를 기반, index+1만 호출하므로 원소의 중복이 존재하지 않는다.
  • 출력 : 경우의 수

주사위 던지기2

  • 재귀 : 해당 원소를 포함한 경우(포함하지 않는 경우는 없다. 주사위는 꼭 주어진 개수만큼 던져야 하니까)
  • 인자 변화 : 사용한 주사위 개수를 기반
    (index를 기반으로 하여 index+1로 호출하였다면 중복된 개수를 셀 수 없다)
  • 출력 : 해당하는 합을 구하는 데에 사용된 원소의 목록

주사위는 최대 7개, 물론 for문 7개로 해결할 수 있다. 하지만 우린 귀찮으니 재귀로 하겠다.
최악의 경우 경우의 수는 $6^7$(279,936)개일 것이다.

함수 : n번째 주사위를 합을 구하는 데 사용하겠다.
인자 : 현재 사용한 주사위 수 n, n번째 주사위까지의 눈의 합 sum
base case : n == N+1일때 더이상 쪼개지지 않는 작업이 된다. (N개까지는 봐야하니까 N+1이다.)
내가 하려는 작업 : sum에 원소를 포함시킨다.
recursive case : n+1번째 주사위에서 내가 볼 수 있는 눈은 1~6이다.

한마디로 말해서 주사위의 개수가 3개인 경우 1 1 1, 1 1 2, 1 1 3, … 6, 6, 6까지 살펴봐야 한다.

유사문제
주사위 던지기1 http://www.jungol.co.kr/bbs/board.php?bo_table=pbank&wr_id=449&sca=20
모든 경우, 중복된 경우, 중복되지 않은 경우를 모두 다룬다

단지번호 붙이기

https://www.acmicpc.net/problem/2667

이 문제 또한 저번 시간에 풀었던 섬의 개수와 매우 유사한 문제이다.
다만 섬의 개수와 다른 점은 다음과 같다.

섬의 개수

  • component 개수 세기
  • 8방향

단지번호 붙이기

  • 한개의 component에 있는 정점들의 개수 세기
  • 4방향
    개수를 세는 연산을 어느 위치에서 해야할지 고민해보자.

함수 : map[y][x]를 방문한다.
인자 : 현재 방문할 y, x
base case : main에서 지도 끝까지 뒤졌을 때 끝난다.(함수 안에는 구현안됨)
내가 하려는 작업 : map[y][x]을 방문한다. visited[y][x]에 방문했다는 표시를 해준다.
recursive case : 4방향을 살펴본 뒤 조건에 맞으면 다음 y, x인자를 설정하여 함수를 호출한다.

유의할 점 : input이 빈칸 없는 1, 0으로 이루어 진 것을 주의하자

유사문제
유기농 배추 https://www.acmicpc.net/problem/1012
영역 구하기 https://www.acmicpc.net/problem/2583

재귀함수 호출(Recursive function call) : 함수가 자기 자신을 호출!

  1. 내가 하려는 작업이 있다
  2. 문제를 잘게 자른다.
  3. 더 이상 자를 수 없는 부분(base) 처리

base case로 더 이상 쪼갤 수 업는 부분을 처리하고
recursive case로 쪼갠 문제를 호출한다.

1182 부분집합의 합

https://www.acmicpc.net/problem/1182

input : 원수의 개수 1<=N<=20
부분집합 원소의 합 S

원소의 개수가 최대 20개일 경우 부분집합 원소의 개수는 $2^20$(1,049,576)개이다.
완전탐색으로 모든 경우의 수를 구한 후 S와 같은 경우의 수를 구할 수 있다.
원소의 값들을 저장한 배열의 index를 인자로 하여 0번 index부터 순차적으로 살펴볼검

arr[index] 원소를 sum에 포함시키는 경우 포함을 시키지 않는 경우

함수 : n번째 원소를 인자로 호출한다.
인자 : 원소의 값을 저장한 배열의 index
base case : index가 N이 되었을 때 더 이상 쪼개지지 않은 작업이 된다.
내가 하려는 작업 : sum에 원소를 포함시킨다, sum에 원소를 포함시키지 않는다.
recursive case : index + 1번째 원소를 인자로 호출한다.

공집합인 경우를 제외하므로 S가 0일 때는 경우의 수 중 한 개를 빼준다.
(S=0일때 공집합일 경우에도 sum == S가 성립하므로)

참고용 : integer return 버전의 재귀함수


4963 섬의 개수

https://www.acmicpc.net/problem/4963

현재 우리가 구현하고 있는 재귀는 DFS(Depth first search : 깊이 우선 탐색) 모양을 가진다.
갈 때까지 가보자는 알고리즘이다. 첨부된 ppt(부분집합의 합 분제)에서 나타나 있는 트리가 맨 마지막 노드까지 내려가는 것처럼!
(뭔말인지 모르겠져 자세한 설명은 그래프 시간에ㅎㅎ)

DFS로 세가지 사실을 알 수 있다.

  1. 탐색
  2. 연결성
  3. component 개수
    (뭔말인지 모르겠져 자세한 설명은 그래프 시간에2ㅎㅎ)

이 문제 같은 경우 component의 개수를 세는 문제이다.
component는 서로 연결되어 있는 뭉탱이로 생각하면 된다.

이 경우 component는 2개이다.
여튼 이 문제에서 이제 재귀함수를 어떻게 구현해야 할지 생각해보자.
재귀함수를 호출하는 것을 방문한다라고 생각을 한다.
그 다음 우리가 다음에 방문하고 싶은 곳을 recursive case에서 인자로 설정하여 방문한다고 생각하면 된다.

함수 : map[y][x]를 방문한다.
인자 : 현재 방문할 y, x
base case : main에서 지도 끝까지 뒤졌을 때 끝난다.(함수 안에는 구현안됨)
내가 하려는 작업 : map[y][x]을 방문한다. visited[y][x]에 방문했다는 표시를 해준다.
recursive case : 8방향을 살펴본 뒤 조건에 맞으면 다음 y, x인자를 설정하여 함수를 호출한다.

recursive case에서 다음 인자를 선택할 조건

nextY를 다음에 방문할 y좌표, nextX를 다음에 방문할 x좌표라고 생각한다.
① map[nextY][nextX]가 육지이다.
② map[nextY][nextX]를 아직 방문하지 않았다. (visited[nextY][nextX]의 값이 0이다.)

Posted on 2015-04-18 20:10 | In Problem Solving | Comments:

Problem

http://www.jungol.co.kr/bbs/board.php?bo_table=pbank&wr_id=497&sca=30

Solution

#
최대 높이는 10억, 최대 막대 갯수는 10만개
당연히 int형 변수는 overflow가 발생할 것이다.

long long -> $-2^{63}$ ~ + $2^{63}$-1
long long s = $w*h$;

이런 연산을 한다고 할 경우
w나 h중 적어도 한 개의 변수는 long long 타입을 가져야 casting을 할 수 있다.
그렇지 않으면 int형 변수로 overflow난 다음에 long long타입의 s변수에 들어가기 때문에 다른 값이 저장된다.

#
a[index]가 정의하는 것은 index번째의 막대 높이
s에 저장되는 원소는 index!
각각 배열이나 스택에 저장되는 요소가 무엇을 뜻하는 지를 정확히 파악하자!

Problem

http://www.jungol.co.kr/bbs/board.php?bo_table=pbank&wr_id=1082&sca=30

Solution

더이상 연산에 필요 없는 원소 (pop하는 원소) : i번째 탑이 볼 수 없는 탑들
즉 i번째 탑을 입력 받았는데 top의 값이 더 작으면 i번째 탑은 top을 볼 수 없다
필요 없으므로 버린다

priority queue, queue, stack을 구현할 때 top()쓸때 존재하는지 아닌지 필수적으로 확인해야 한다 (dequeue error)
